What a great sentry! But the shield needed a bit of cleaning.

Description

The sentry.io application is a code monitoring system. It's organized through projects. Each project has issues generated by exceptions (for example) in your code. The sentry web application has an API to manage these functionalities. The API requires authentication with an auth token. The token can have various types of privileges. You set them when you create the token. By combining these permissions, you can operate certain actions in the API.

The aforementioned functionality requires CORS: The sentry web application is not aware of the domain you will send issues from. Therefore, sentry requires a flexible CORS configuration.

The problem

The CORS configuration was too flexible. It asserted the domain the request came from ended in sentry.io. To exploit this, an attacker could buy a domain: attacker-sentry.io and host js code to perform a CORS request with credentials.

To escalate this behavior, an attacker could force his sentry target to fetch the victim's auth tokens. Those tokens could be sent to an attacker controlled server, and used to manage the organization: An attacker could create and delete projects (for example).

Report

Reporting this issue was fantastic! The sentry.io team rewarded me with a good bounty and a bonus. They fixed the issue very quickly. It was a privilege to work with them!

Conclusion

It's important to consider the context of the CORS configurations, and how the CORS implementation can be abused by an attacker. Allow only what's needed. Deny everything else.
